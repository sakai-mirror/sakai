Coding Standards

This document describes a list of coding conventions that are required for code submissions to the project. By default, the coding conventions for most Open Source Projects should follow the existing coding conventions in the code that you are working on. For example, if the bracket is on a new line, then you should write all your code to use that same convention.

If you commit code that does not follow these conventions, you are responsible for also fixing your own code.

Below is a list of coding conventions that are specific to Navigo, everything else not specifically mentioned here should follow the official Sun Java Coding Conventions and The Elements of Java Style (ISBN: 0521777682).

  1. Brackets should begin and end on a new line and should exist even for one line statements. Examples: 
if(foo)
{
    // code here
}

try
{
    // code here
}
catch(Exception bar)
{
    // code here
}
finally
{
    // code here
}

while(true)
{
    // code here
}
  1. Though it's considered acceptable to include spaces inside parentheses, the preference is to not include them. 
if(foo)    // preferred

or

if ( foo ) // acceptable
  1. 2 space indent. NO tabs . Period. We understand that many developers like to use tabs, but the fact of the matter is that in a distributed development environment where diffs are sent to the mailing lists by both developers and the version control system (which sends commit log messages), the use of tabs makes it impossible to preserve legibility.
     
  2. Unix linefeeds for all .java source code files.  Please set your editor appropriately.
     
  3. JavaDoc MUST exist on all methods. If your code modifications use an existing class/method/variable which lacks JavaDoc, it is required that you add it. This will improve the project as a whole.
     
  4. The Navigo License MUST be placed at the top of each and every file.
     
  5. If you contribute to a file (code or documentation), add yourself to the authors list at the top of the file.  The class or interface creator should be listed first.  For java files the preferred Javadoc format is: 
@author <a href="mailto:user@domain.com">John Doe</a>
  1. All .java files should have a @version tag like the one below. 
@version $Id: code-standards.html,v 1.1.1.1 2004/07/28 21:31:45 rgollub.stanford.edu Exp $
  1. Import statements must be fully qualified for clarity. 
import java.util.ArrayList;
import java.util.Hashtable;

import org.navigo-project.foo.Bar;
import org.navigo-project.bar.Foo;

And not 

import java.util.*;
import org.apache.foo.*;
import org.apache.bar.*;
  1. Log4j logging statements should be included as classes are built and not as an afterthought.  Once the application is moved into production, the output from log4j will be the primary troubleshooting and debugging method.
     
  2. Junit units tests should be written along with classes and not as an afterthought.  These tests should be integrated in a test suite which will be performed and evaluated during each major and minor release.  These junit tests should map very closely back to requirements gathered in UML documentation.
     
  3. Passwords or usernames should NEVER be checked into version control.

Design Standards

  1. UML will be used to document the requirements of the application.  Functionality will not be added to the application without corresponding UML documentation.
  2. UML documentation should include detailed Use Case and Class Diagrams at a minimum.  Inclusion of activity diagrams is preferred.
  3. The application will be comprised of three distinct tiers: Data, Business, and Presentation.  Classes should be designed to fit neatly into one of these layers.
