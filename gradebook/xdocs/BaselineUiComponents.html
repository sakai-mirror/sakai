Baseline Gradebook UI Components

by Ray Davis - 16 February

Each top item in the list names a page area that seems as if it might work as an entity outside any single page, or that seems to require something that normal JSF can't handle. (I don't bother to list UI elements that are only associated by CSS class, text resource, or normal JSF tags.) I briefly list each item's subcomponents and likely runtime parameters, note what I've found in the way of existing documentation or support, and suggest what to do about it.

  1. AppMenu : TaskTextLinks + (optional) CurrentTask
    • Style Guide – This may be the "Intra-tool Navigation" (with navIntraTool CSS class) listed in the table of contents of the 1.3 document, but it's hard to tell since that section is blank.
    • Samigo – Implemented in-line on each page rather than as include-fragment or component. CVS includes a Navigation Map tag, but it doesn't seem to support a "current selection" argument.
    • APPROACH
      1. Suggest for Style Guide.
      2. Start with a JSP 2 tag. (The AppMenu + Title at the top of each page looks pretty Tiles-like, but right now I don't see enough payback from Tiles given our small number of pages and the otherwise wide variation between them.)
      3. Possibly move to JSF tag later. (MyFaces HtmlPanelNavigation shows an approach for applications in which the user will always be in a menu context: Define AppMenu as a collection of HtmlCommandLink components. When a command link is activated, its "broadcast(event)" method deactivates other child command links and sets its own status to active. The command link's active state then determines its rendering. In the Baseline Gradebook, however, not all task pages appear in the AppMenu. Assignment Grades, Course Grades, Edit Assignment, Remove Assignment, and Student View are all reached in other ways.)

  2. PageContent : Title + Subtitle + Instructions + Messages + PageSpecific
    • APPROACH
      1. All of these (except Title, Message, and "PageSpecific") are optional, which is one reason I'm iffy about going to Tiles just to consolidate them. It might be worth using JSP inclusion to set the page content structure and class defintions up, or, given the small number of pages, we might just copy-and-paste as needed.

  3. Messages : Passed from backing bean to next page; thrown away once displayed.
    • APPROACH
      1. Check MyFaces standard JSF Message support to see whether we can use it directly, can build on it (as with the Pilot renderer), or have to roll our own message bean in session scope.

  4. PagingTable : SearchFilter + PagingControls + SortableTable
    • Style Guide – Included as "List", except that instead of a search filter there's an unlikely sounding "View" dropdown menu.
    • Samigo – A Pager tag is in CVS, but I haven't seen it in action in the demos. Unknown amount of work needed.
    • MyFaces – Provides similar functionality with a combination of two tags: an extended dataTable and a dataScroller that refers to it. No automatic support for range selection menu, search filtering, or table summary. Unknown amount of work to make the format match our wireframes—facets are used for the controls.
    • APPROACH
      1. Suggest changes to Style Guide.
      2. The application UI will start with non-paging tables. Tag options can be investigated in parallel.
      3. Compare our requirements to Samigo and try to get a tag that will support us both.
      4. Depending on Samigo tag schedule, check basic paging workflow with the MyFaces control and search filtering workflow with a separate input area.

  5. PagingControls : BeginningButton + PreviousButton + RangeSelect + NextButton + EndButton + CurrentRange
    • Style Guide – Described (and improved in 1.3) as "List Navigator".
    • MyFaces – dataScroller tag.

  6. ItemDetail : DetailNavigation + DetailDescription
    • Style Guide – Describes this as an "Item Summary page". I doubt it can be turned into a single component.

  7. DetailNavigation : Previous + ReturnTo + Next
    • Style Guide – Describes this as a "List Navigator".
    • APPROACH
      1. Since we only have one page that needs this, just embed the logic in that page. It should be made into a standard tag later for use by other tools.

  8. SortableTable : Headers + CellComponents (usually text or numbers, input field for points) + SortColumn + SortDirection + ColumnSortingComparison (case-insensitive alphabetic; numeric; date; boolean; special [e.g., "Final Grade" always appears as if it has the latest due date, even after undated assignments]) + DynamicallyDefinedColumns (needed when the columns are assignments); optionally one more column than headers
    • Style Guide – Describes this as a "List" or an "Actionable List" depending on whether all cells are readonly or not. Unspecified support for column sorting comparisons, dynamically defined columns, and dangling columns. Paging position after sort changes or range changes is unspecified.
    • Samigo – The basic functionality has started to show up, with some bugs and limitations, but doesn't seem to be componentized yet. Instead, each sortable header is specified in the JSP as a separate commandLink in a panelGroup.
    • MyFaces – Provides similar functionality with an extended dataTable and commandSortHeader tags. Need further research to find out support for multiple sorting comparisons (might need to do it by subclassing cell values to implement the Comparable interface) and dynamically defined columns.
    • APPROACH
      1. Suggest additions to Style Guide.
      2. The current Samigo approach is very bulky. Try to componetize the functionality, using MyFaces (and maybe Oracle ADF) as inspiration, including our "advanced" requirements.
      3. Depending on tag implementation schedule, we may build on the MyFaces components to test basic workflow.
    • WARNING – Given the difficulty of building complex dynamic components in JSF and the difficulty of building on existing JSF renderers, this is where we're most likely to fail to deliver even the reduced functionality of the baseline Sakai 2.0 gradebook. If, as seems possible to me, we're unable to satisfy all requirements within the schedule, I'm afraid that some usability and functionality will have to be unpredictably sacrificed to our choice of presentation framework. I also worry that we might not be able to make a JSP 2 implementation coexist peacefully with the JSF lifecycle; at any rate, by the time we feel forced to try that step, it would probably be too late to implement it. (I may look for a suitable JSP 2 tag, just in case.)

  9. DateSelect : OptionalCheckbox (could be implemented as a different component in same form area) + MonthDayYearSelect + CalendarSelectPopup
    • Style Guide – Describes this as "Date/Time Entry".
    • Samigo – Has a DatePicker tag.
    • MyFaces – Has inputDate and inputCalendar tags.
    • APPROACH
      1. Start with the Samigo DatePicker. If we have problems (unlikely), we can temporarily fall back to the MyFaces tag.
